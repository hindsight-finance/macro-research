## 1. What is an HMM?

A Hidden Markov Model (HMM) models a system where the true **state** is unobservable ("hidden"), but you observe **emissions** (data) that are probabilistically generated by that hidden state.

```
Hidden States:    S₁ ──→ S₂ ──→ S₁ ──→ S₁ ──→ S₃ ...
                   ↓       ↓       ↓       ↓       ↓
Observations:     O₁      O₂      O₃      O₄      O₅
```

The key insight: **you never observe the regime directly** — you only observe price data, and the model infers which hidden state most likely generated that data.

### Formal Definition

Given `N` hidden states and observed data `O = {o₁, o₂, ..., oₜ}`, an HMM is defined by:

| Component | Notation | Description |
|-----------|----------|-------------|
| States | `S = {s₁, ..., sₙ}` | Latent regimes (e.g., bull/bear/volatile) |
| Transition matrix | `A[i,j] = P(sⱼ \| sᵢ)` | Probability of switching from state i to j |
| Emission distribution | `B(sᵢ)` | Distribution of observations given state i |
| Initial distribution | `π` | Starting state probabilities |

### The Two Core Assumptions

**1. Markov property** — current state depends only on the previous state:
```
P(sₜ | s₁, s₂, ..., sₜ₋₁) = P(sₜ | sₜ₋₁)
```

**2. Conditional independence of observations** — current observation depends only on current state:
```
P(oₜ | s₁..sₜ, o₁..oₜ₋₁) = P(oₜ | sₜ)
```

## 2. Core Components

### Transition Matrix (Regime Persistence)

The `N × N` matrix `A` encodes how regimes persist and switch. For a 3-state model:

```
         → Bull   Bear   Chop
Bull   [ [0.92,  0.05,  0.03],
Bear     [0.04,  0.91,  0.05],
Chop     [0.10,  0.10,  0.80] ]
```

- High diagonal values → regimes are sticky (persist for many bars)
- Off-diagonal values → transition probabilities between regimes
- Expected regime duration = `1 / (1 - Aᵢᵢ)` bars

### Emission Distribution

Each state has its own distribution over observed data. Common choices:

| Type | When to Use |
|------|-------------|
| `GaussianHMM` | Single return series; simple, fast |
| `GMMHMM` | Fat-tailed returns; mixture of Gaussians per state |
| Multivariate Gaussian | Multiple features (returns + vol + breadth) |
| Custom | Non-parametric or domain-specific |

For a 2-state model on returns, the model might learn:
- **State 0 (Bull)**: mean = +0.05%/bar, σ = 0.8%
- **State 1 (Bear)**: mean = -0.12%/bar, σ = 2.1%

## 3. The Three Algorithms

### Viterbi (Decoding) — "What regime are we in?"

Finds the most likely **sequence** of hidden states given all observations:

```python
states = model.predict(observations)
# → [0, 0, 0, 1, 1, 0, 2, 2, ...]  discrete regime labels
```

**Problem for trading**: uses all data including future bars — look-ahead bias.

### Forward-Backward (Smoothing) — "What's the probability of each regime?"

Computes `P(sₜ | all observations)` — a soft probability across all states:

```python
posteriors = model.predict_proba(observations)
# → [[0.92, 0.06, 0.02],   ← 92% bull, 6% bear, 2% chop at t=1
#    [0.85, 0.12, 0.03],
#    ...]
```

**Problem for trading**: still uses future data ("backward" pass). Better for research/backtesting.

### Forward Algorithm Only (Filtering) — "Real-time regime probability"

Computes `P(sₜ | o₁..oₜ)` using only past and current data. Safe for live trading:

```python
# Online filtering: only conditions on past data
log_prob, forward_probs = model.score_samples(observations[:t+1])
current_regime_probs = forward_probs[-1]  # only last row
```

### Baum-Welch (Learning) — "Fit the model to data"

Expectation-Maximization (EM) algorithm that estimates `A`, `B`, `π` from data with no labeled training:

```python
model = hmm.GaussianHMM(n_components=3, n_iter=100)
model.fit(returns.reshape(-1, 1))
# Model now has learned: transition matrix, emission params, initial probs
```

## 4. HMM for Regime Classification

### Minimal Example

```python
from hmmlearn import hmm
import numpy as np

# Fit a 3-state Gaussian HMM to log returns
returns = np.log(df['close'] / df['close'].shift(1)).dropna()
model = hmm.GaussianHMM(n_components=3, covariance_type="full", n_iter=200)
model.fit(returns.values.reshape(-1, 1))

# Decode most likely regime sequence (research use — has look-ahead)
states = model.predict(returns.values.reshape(-1, 1))

# Regime probabilities per bar (research use)
posteriors = model.predict_proba(returns.values.reshape(-1, 1))

# Inspect what the model learned
for i in range(model.n_components):
    print(f"State {i}: mean={model.means_[i][0]:.4f}, vol={np.sqrt(model.covars_[i][0,0]):.4f}")

print("Transition matrix:")
print(np.round(model.transmat_, 3))
```

### Multi-Feature HMM

Feed multiple signals as multivariate emissions rather than raw returns alone:

```python
# Feature matrix: each row is one bar
features = np.column_stack([
    returns,              # Return
    realized_vol,         # Volatility
    adx_signal,           # ADX trend strength
    hurst_rolling,        # Hurst rolling estimate
])

model = hmm.GaussianHMM(n_components=3, covariance_type="full", n_iter=200)
model.fit(features)
```

### Key Design Decisions

| Decision | Options | Notes |
|----------|---------|-------|
| Number of states | 2–4 | More states overfit; use AIC/BIC |
| Emission type | Gaussian, GMM, custom | GMM handles fat tails better |
| Features | Returns, vol, indicators | More features → more data needed |
| Algorithm | Viterbi, filtering, smoothing | Filtering only for live signals |
| Initialization | Random, k-means, domain | Domain init speeds convergence |

### Initialization Sensitivity

Baum-Welch converges to local optima. Mitigation strategies:
1. Multiple random restarts, keep highest log-likelihood
2. Initialize means with k-means clustering
3. Initialize with domain knowledge (high/low vol states)

```python
best_model, best_score = None, -np.inf
for seed in range(20):
    model = hmm.GaussianHMM(n_components=3, n_iter=100, random_state=seed)
    model.fit(features)
    score = model.score(features)
    if score > best_score:
        best_score, best_model = score, model
```

## 5. Comparison: HMM vs This System

### Architecture Overview

| Dimension | This System (StateDetector) | HMM |
|-----------|-----------------------------|-----|
| **Paradigm** | Rule-based ensemble | Probabilistic generative model |
| **Learning** | Manual weights & thresholds | Learns from data (Baum-Welch) |
| **States** | Predefined: STRONG_TREND, WEAK_TREND, CONSOLIDATION, CHOPPY, UNCERTAIN | N states discovered empirically |
| **Regime persistence** | No memory — each call is independent | Explicit via transition matrix |
| **Uncertainty** | `\|composite - 0.5\| × 2` (distance from neutral) | True posterior probability |
| **Direction** | Simple price change threshold (0.1%) | Encoded in emission means |
| **Adaptability** | Hurst-based dynamic weight shift | Fixed post-training (unless retrained) |
| **Transparency** | Full: see exactly which indicator drove signal | Partial: states must be interpreted post-hoc |
| **Training data needed** | None | Yes — needs representative history |

---

### State Classification

**This system** uses deterministic threshold rules on a weighted composite signal:

```
composite ≥ 0.70  →  STRONG_TREND
composite ≥ 0.55  →  WEAK_TREND
composite ≥ 0.45  →  WEAK_TREND   (borderline)
composite ≥ 0.30  →  CONSOLIDATION
composite  < 0.30  →  CHOPPY
confidence < 0.30  →  UNCERTAIN    (overrides)
```

**HMM** outputs a posterior distribution — the model doesn't classify via thresholds, it assigns probabilities across all states simultaneously:

```
P(BULL  | data) = 0.72
P(BEAR  | data) = 0.21
P(CHOP  | data) = 0.07
```

The argmax of these gives a hard label, but the distribution itself is richer information. A 72% bull vs. 90% bull have very different downstream implications.

---

### Regime Persistence

**This system** has no memory between calls. Each `detect()` call is stateless — the classification at t=100 does not depend on the classification at t=99 in any structural way (only through the indicator calculations which use the same window).

**HMM** explicitly models persistence through the transition matrix diagonal. If a model learns `A[BULL,BULL] = 0.95`, it means: conditional on being in the bull regime, there is a 95% chance of remaining in it next bar. This creates:
- **Natural smoothing** — noisy single bars won't flip the regime
- **Transition signals** — off-diagonal probability spikes are early warning of regime change
- **Expected duration** — `1/(1-0.95) = 20 bars` expected time in bull regime

---

### Dynamic Weight Adaptation vs. Non-Stationarity

**This system** handles non-stationarity via Hurst-based dynamic weight shifting:

```python
# When Hurst > 0.5 (trending regime detected):
# → upweight ADX, MSS (trend-measuring indicators)
# → downweight DRA, ATR/Range, IRR (consolidation-measuring)
adjustment = alpha * tanh(kappa * (hurst - 0.5))
```

This is a principled, continuous adaptation mechanism with no training required. It uses the Hurst signal as a meta-feature to self-calibrate the ensemble.

**HMMs** assume a stationary transition matrix by default. They do not adapt weights or parameters during inference. Workarounds:
- **Regime-switching HMM**: separate models per macro state
- **Input conditioning**: include Hurst as a feature in the emission distribution
- **Rolling retraining**: refit the model periodically on a sliding window
- **Non-homogeneous HMM**: time-varying transition probabilities (complex)

**Verdict**: On non-stationarity, this system's dynamic weighting is genuinely more sophisticated than a standard HMM.

---

### Feature Extraction

**This system** runs 7 specialized indicators, each designed to capture a distinct aspect of market structure:

| Indicator | What It Captures |
|-----------|-----------------|
| ADX | Directional movement strength + DI persistence |
| MSS | Slope consistency across multiple time scales |
| IRR | Intraperiod reversion (wick vs body structure) |
| DRA | Opening range breakout and persistence |
| ATR/Range | Volatility relative to net directional progress |
| SPD | Swing point count (choppiness measure) |
| Lag/Hurst | Statistical return persistence |

Each is domain-derived and interpretable. The features are the result of significant domain knowledge.

**HMM** takes raw inputs (returns, realized vol, etc.) and learns the statistical patterns from data. It does not "understand" swing points or opening ranges — it learns that certain return/vol combinations tend to cluster together. Using the 7 indicator signals *as inputs to an HMM* is a meaningful hybrid (see Section 8).

---

### Confidence Estimation

**This system**: `confidence = |composite_signal - 0.5| × 2`
- Max at extremes (0 or 1), min at neutral (0.5)
- Purely a function of the composite signal
- Not calibrated to any ground truth

**HMM**: `confidence = max(posterior_probs)`
- Derived from probability model fit to data
- Naturally accounts for ambiguous overlapping emission distributions
- Can be calibrated (Platt scaling, isotonic regression)
- Provides full distribution, not just a scalar

---

### Look-Ahead Risk

**This system**: Processes a complete session window in one call. If used for research/backtesting on a full historical series, it does not introduce future look-ahead *within the session* — each call sees only that session's bars. However, parameter tuning (thresholds, weights) on historical data is a form of look-ahead.

**HMM**: The forward-backward algorithm and Viterbi both use future data during smoothing. For live signals you must use the **forward algorithm only**. This is a concrete implementation risk — using `model.predict()` in backtesting introduces look-ahead.

## 6. Where HMM Would Add Value

### 1. Regime Transition Probability

An HMM's off-diagonal transition probabilities give you a direct signal for regime change. There is no equivalent in the current system.

```python
# "How likely is a regime change happening right now?"
transition_prob = 1.0 - posteriors[t, current_state]
# Or: probability mass flowing into the new state
```

This could be a valuable feature in its own right — a "regime instability" signal.

### 2. Duration Modeling

From the learned transition matrix, you can derive:
- Expected time remaining in current regime
- Probability of still being in this regime after N more bars
- Historical average regime durations per state

This is structural information the current system cannot produce.

### 3. Learning Thresholds from Data

The current system's classification thresholds (`0.30`, `0.45`, `0.55`, `0.70`) and weights (`adx: 0.18`, `mss: 0.24`, etc.) were set by judgment and calibrated on examples. An HMM would learn the natural clusters from data without imposing these.

### 4. State Coherence Across Sessions

The current system treats each session independently. An HMM trained on a multi-session history would learn cross-session regime dynamics — e.g., that a strong trend session tends to be preceded by a consolidation session.

### 5. Probabilistic Strategy Allocation

With full posterior probabilities, you can allocate strategy exposure proportionally:

```python
# Instead of binary "trending / not trending":
trend_exposure = posteriors[t, bull_state]  # 0.72 → 72% trend strategy allocation
```

## 7. Where This System is Superior

### 1. No Training Data Required

The current system works on the first session of data. An HMM needs sufficient historical data to estimate parameters reliably. For a new instrument or session type, the HMM cannot be deployed until enough history accumulates.

### 2. Interpretable Signal Decomposition

When StateDetector returns `CHOPPY`, you can inspect `result.signals` to see:
- ADX = 0.31 (weak directional movement)
- SPD = 0.28 (high swing count)
- IRR = 0.25 (heavy intraperiod reversion)

An HMM produces a probability but cannot tell you which aspect of the data drove it. This matters for debugging, trust-building, and regime understanding.

### 3. Multi-Perspective Robustness

The 7 indicators are designed to be complementary and partially decorrelated — price action (MSS, ADX), statistical (Lag/Hurst), range-based (DRA, ATR), and structural (SPD, IRR). This ensemble approach guards against any single indicator misfiring.

An HMM with raw returns as input only sees one view of the data.

### 4. Hurst-Adaptive Weighting

The dynamic weight adjustment based on the Hurst exponent is genuinely novel — using a statistical measure of return persistence to reweight the ensemble in real time, without any training. A standard HMM has no equivalent mechanism.

### 5. Session-Aware Configuration

Different bar sizes, indicator configurations, and expected behaviors for `1pm-3pm`, `3pm-3:50pm`, and `3:50pm-4pm` are explicitly encoded. An HMM would need to be trained separately per session or have session as an input feature.

### 6. No Initialization Sensitivity

The current system is fully deterministic given the same input data. HMMs may converge to different local optima from different random initializations, requiring multiple restarts and careful validation.

### 7. Graceful Degradation

If 2 of 7 indicators fail, the system automatically reweights the remaining 5 and continues. An HMM has no equivalent mechanism — if a feature is missing, the emission distribution cannot be evaluated.

## 8. Hybrid Integration Paths

The most powerful approach is not to replace the current system with an HMM, but to combine them.

### Path A: HMM on Top of Existing Signals (Recommended)

Use the 7 indicator signals as **features for a multivariate HMM** rather than raw price data. The HMM then learns regime-specific distributions over the indicator space:

```python
# Build feature matrix from existing indicator signals
features = np.column_stack([
    adx_signals,    # Already normalized 0-1
    mss_signals,    # Already normalized 0-1
    irr_signals,    # Already normalized 0-1
    dra_signals,    # Already normalized 0-1
    atr_signals,    # Already normalized 0-1
    spd_signals,    # Already normalized 0-1
    lag_signals,    # Hurst exponent
])

model = hmm.GaussianHMM(n_components=4, covariance_type="full")
model.fit(features)
```

**Advantages**:
- Preserves domain knowledge in the features
- HMM learns the natural clusters in indicator-space rather than imposing thresholds
- Transition matrix captures regime persistence not present in current system
- Posterior probabilities replace the `|composite - 0.5| × 2` confidence estimate

### Path B: HMM for Transition Detection Only

Keep the current StateDetector as the primary classifier. Add an HMM layer that uses the StateDetector's output sequence to model regime transitions:

```python
# Run StateDetector on rolling windows
state_sequence = [detector.detect(window).state for window in windows]

# Encode states as integers
encoded = [state_map[s] for s in state_sequence]

# Fit HMM on state sequence (Multinomial emissions)
meta_model = hmm.MultinomialHMM(n_components=2)  # "stable" vs "transitioning"
meta_model.fit(encoded)

# Use transition probabilities as a "regime instability" signal
```

### Path C: Hurst-Conditioned HMM

Incorporate the Hurst signal — which already drives dynamic weighting — as a conditioning variable for the transition matrix. When Hurst > 0.6, reduce the probability of transitioning out of the trending state:

```python
def hurst_adjusted_transitions(base_transmat, hurst):
    """Adjust transition stickiness based on Hurst exponent."""
    adjustment = np.tanh(4.0 * (hurst - 0.5))  # -1 to +1
    # Increase diagonal (persistence) when trending, decrease when mean-reverting
    adjusted = base_transmat.copy()
    for i in range(len(adjusted)):
        adjusted[i, i] += 0.1 * adjustment
    # Renormalize rows
    adjusted = adjusted / adjusted.sum(axis=1, keepdims=True)
    return adjusted
```

### Path D: Validation / Disagreement Signal

Run both systems in parallel. When they agree, confidence is high. When they disagree, flag the bar as UNCERTAIN regardless of individual confidences:

```python
hmm_state = hmm_model.predict(features[-1:])[0]
detector_state = detector.detect(df).state

if hmm_state_map[hmm_state] != detector_state:
    # Disagreement — regime is ambiguous
    final_state = "UNCERTAIN"
```

**Bottom line**: The current StateDetector is a well-engineered, domain-driven ensemble that does things HMMs structurally cannot (Hurst-adaptive weighting, multi-perspective diagnostics, session-awareness). The primary gaps — regime persistence modeling and principled posterior probabilities — are where an HMM layer would add the most value. Path A (HMM on top of indicator signals) is the most natural integration and would leverage the existing work rather than replace it.
